# Data Model: Document Upload and Management

## Document

- Purpose: Represents a stored file plus the metadata and authorization anchors needed to browse, search, preview, download, replace, and delete it.
- Primary Key: `DocumentId` (int)
- Core Fields:
  - `Title` (string, required, max 255)
  - `Description` (string, optional, max 2000)
  - `Category` (string, required, max 100; stores text values)
  - `OriginalFileName` (string, required, max 255)
  - `StorageKey` (string, required, max 500; relative private storage path)
  - `FileType` (string, required, max 255)
  - `FileSizeBytes` (long, required)
  - `UploadedByUserId` (int, required)
  - `ProjectId` (int, optional)
  - `TaskId` (int, optional)
  - `CreatedDateUtc` (DateTime, required)
  - `UpdatedDateUtc` (DateTime, required)
- Relationships:
  - Many documents to one uploader (`User`)
  - Many documents to zero-or-one `Project`
  - Many documents to zero-or-one `TaskItem`
  - One document to many `DocumentShare` records
  - One document to many `DocumentActivityRecord` records
  - One document to many `DocumentTag` values
- Validation Rules:
  - Title and category are mandatory.
  - File type must be one of the supported upload types.
  - File size must be greater than zero and no more than 25 MB.
  - StorageKey must be generated by the application and never derived directly from user input.
  - If `TaskId` is populated and the task has a project, `ProjectId` must match the task's project.
- Lifecycle / State Transitions:
  - Selected -> Stored when file save succeeds and metadata record commits.
  - Stored -> Replaced when a new file succeeds and the record updates.
  - Stored -> Deleted when a confirmed delete removes the record and binary.

## DocumentTag

- Purpose: Represents a single searchable tag value attached to a document.
- Primary Key: `DocumentTagId` (int)
- Core Fields:
  - `DocumentId` (int, required)
  - `TagValue` (string, required, max 100)
- Relationships:
  - Many tags to one document
- Validation Rules:
  - Tag values are trimmed, case-normalized for search, and unique per document.

## DocumentShare

- Purpose: Grants access outside direct ownership via a specific recipient user or the app's existing department/team scope.
- Primary Key: `DocumentShareId` (int)
- Core Fields:
  - `DocumentId` (int, required)
  - `SharedByUserId` (int, required)
  - `SharedWithUserId` (int, optional)
  - `SharedWithDepartment` (string, optional, max 100)
  - `CreatedDateUtc` (DateTime, required)
- Relationships:
  - Many shares to one document
  - Many shares to one sharing user
  - Optional share target to one recipient user
- Validation Rules:
  - Exactly one recipient target is allowed: either `SharedWithUserId` or `SharedWithDepartment`.
  - Department shares must use an existing department/team scope value already present on users.
  - Duplicate active shares for the same target should be prevented.

## DocumentActivityRecord

- Purpose: Provides the auditable event stream required for upload, download, preview, replace, delete, and share reporting.
- Primary Key: `DocumentActivityRecordId` (int)
- Core Fields:
  - `DocumentId` (int, required)
  - `ActorUserId` (int, required)
  - `ActivityType` (string or enum, required)
  - `OccurredAtUtc` (DateTime, required)
  - `Details` (string, optional, max 1000)
- Relationships:
  - Many activity records to one document
  - Many activity records to one actor user
- Validation Rules:
  - Activity type must be one of the allowed audited actions.
  - Records are append-only.

## Notification Extensions

- Purpose: Extend the existing notification model to describe document-specific events without creating a separate notification subsystem.
- Planned Additions:
  - `NotificationType.DocumentShared`
  - `NotificationType.DocumentAddedToProject`

## Authorization Model Notes

- Owners can view, update metadata, replace, share, and delete their own documents.
- Current project members can upload project documents and view project documents they are entitled to access.
- Team leads can manage documents uploaded by users in their own department/team scope only.
- Project managers can manage documents within projects they manage.
- Administrators can access all documents and reports.

## Derived Query Shapes

- My Documents: documents where `UploadedByUserId == requestingUserId`
- Shared with Me: documents with a share targeting the requesting user or their department/team scope
- Project Documents: documents with matching `ProjectId` and project-membership authorization
- Recent Documents Widget: latest five accessible documents ordered by `CreatedDateUtc` descending
